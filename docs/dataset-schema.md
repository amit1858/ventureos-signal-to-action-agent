# Dataset Schema: Signal-to-Action Agent

## Overview

All data in the Signal-to-Action Agent MVP is **synthetic** and generated by `services/api/data/generate_synthetic_data.py`. The dataset models real-world enterprise customer signals without containing any confidential, proprietary, or PII data. Data is stored in CSV files for simplicity (accounts.csv, signals.csv, notes.csv) and loaded into memory during API startup.

**Cardinality:** ~50 synthetic accounts, ~200+ signals, ~75 notes.

---

## Schema: Account

Represents a customer account (SMB business) with spend, usage, engagement, and renewal metadata.

| Field | Type | Range/Units | Description |
|-------|------|-------------|-------------|
| `account_id` | string | UUID (e.g., `ACC-001`) | Unique account identifier |
| `account_name` | string | — | Synthetic business name (e.g., "TechStart Solutions", "CloudBridge Inc.") |
| `industry` | string | enum | One of: `Technology`, `Healthcare`, `Retail`, `Finance`, `Manufacturing`, `Education` |
| `segment` | string | enum | Account tier: `SMB` (small/medium business), `Mid-Market`, `Enterprise` |
| `region` | string | enum | Geographic region: `North America`, `Europe`, `APAC`, `LATAM` |
| `current_month_spend` | float | USD | Monthly recurring revenue (MRR) in current month (e.g., 4500.00) |
| `previous_month_spend` | float | USD | MRR in previous month; used to compute spend trend |
| `product_usage_score` | int | 0–100 | Normalized product adoption/feature usage score (higher = more engaged) |
| `engagement_score` | int | 0–100 | Composite engagement: logins, feature clicks, session time (higher = more active) |
| `support_risk_score` | int | 0–100 | Support ticket severity/frequency risk (higher = riskier; 0 = no issues) |
| `campaign_response_score` | int | 0–100 | Marketing campaign engagement (email opens, clicks, webinar attendance) |
| `last_contact_days` | int | days | Days since last seller touchpoint (call, email, meeting). Higher = stale. |
| `renewal_days` | int | days | Days until contract renewal. Negative values = overdue renewal. |
| `growth_potential_score` | int | 0–100 | Propensity to expand seats/products based on firmographic + usage patterns |

**Example Row:**
```csv
ACC-023,DataViz Partners,Technology,SMB,North America,5200.00,4800.00,72,68,25,81,18,90,78
```

**Interpretation:** DataViz Partners is a North American SMB tech company with growing spend (+8.3% MoM), strong usage (72/100), good engagement (68/100), low support risk (25/100), high campaign response (81/100), was contacted 18 days ago, has renewal in 90 days, and high growth potential (78/100).

## Schema: Signal

Represents a discrete customer event or behavioral indicator from a source system (CRM, billing, support, telemetry, marketing).

| Field | Type | Range/Units | Description |
|-------|------|-------------|-------------|
| `signal_id` | string | UUID (e.g., `SIG-001`) | Unique signal identifier |
| `account_id` | string | foreign key | Links to `Account.account_id` |
| `signal_type` | string | enum | Event category (see below) |
| `signal_date` | string | ISO 8601 date | When signal occurred (e.g., `2026-05-28`) |
| `signal_strength` | float | 0.0–1.0 | Importance weight (1.0 = critical, 0.3 = weak) |
| `signal_description` | string | — | Human-readable event detail (e.g., "Declined payment retry #2") |
| `source_system` | string | enum | Originating system (see below) |
| `positive_or_negative` | string | enum | Polarity: `positive`, `negative`, `neutral` |

### `signal_type` Examples

| Signal Type | Description | Typical Polarity |
|-------------|-------------|------------------|
| `usage_spike` | Sudden increase in product usage (e.g., +50% logins) | Positive |
| `usage_drop` | Sudden decrease in usage (e.g., -40% feature engagement) | Negative |
| `support_ticket` | Support case opened (severity encoded in strength) | Negative |
| `payment_delay` | Invoice overdue or payment retry failed | Negative |
| `campaign_click` | Clicked email link, attended webinar, downloaded asset | Positive |
| `login_drop` | No logins in 14+ days (dormancy signal) | Negative |
| `renewal_upcoming` | Contract renewal window opened (<90 days) | Neutral |
| `nps_response` | Net Promoter Score survey submitted | Positive/Negative (depends on score) |
| `feature_adoption` | Activated a new premium feature or integration | Positive |
| `pricing_inquiry` | Asked about pricing, upsell, or expansion | Positive |
| `churn_intent` | Downgrade request, cancellation inquiry, competitor mention | Negative |

### `source_system` Examples

| Source System | Description |
|---------------|-------------|
| `CRM` | Salesforce, HubSpot (contact logs, opportunity stage changes) |
| `Billing` | Stripe, Zuora (invoices, payment failures, subscription changes) |
| `Support` | Zendesk, Intercom (tickets, CSAT scores, escalations) |
| `Telemetry` | Product analytics (logins, feature usage, session duration) |
| `Marketing` | Marketo, Pardot (campaign engagement, email clicks, webinar attendance) |

**Example Row:**
```csv
SIG-042,ACC-023,campaign_click,2026-05-25,0.75,Clicked 'Q2 Product Updates' webinar link,Marketing,positive
```

**Interpretation:** On May 25, DataViz Partners (ACC-023) clicked a webinar link from a marketing campaign—a moderately strong (0.75) positive signal from the Marketing system.

## Schema: Note

Represents free-text seller/CSM notes or CRM activity logs for an account.

| Field | Type | Range/Units | Description |
|-------|------|-------------|-------------|
| `account_id` | string | foreign key | Links to `Account.account_id` |
| `note_type` | string | enum | One of: `call`, `email`, `meeting`, `internal_note`, `support_escalation` |
| `note_text` | string | — | Seller/CSM observation (e.g., "Customer mentioned interest in API tier upgrade") |
| `created_date` | string | ISO 8601 date | When note was created |

**Example Row:**
```csv
ACC-023,call,"Discussed Q2 roadmap; customer interested in advanced analytics tier.",2026-05-10
```

## Schema: Recommendation

Returned by `POST /api/recommendations`. Each recommendation represents a prioritized account with evidence, action, and governance metadata.

| Field | Type | Description |
|-------|------|-------------|
| `recommendation_id` | string | Unique ID (e.g., `REC-20260528-001`) |
| `account_id` | string | Links to Account |
| `account_name` | string | Denormalized for convenience |
| `priority_rank` | int | Rank position (1 = highest priority) |
| `priority_score` | float | Deterministic composite score (0–1) |
| `priority_reason` | string | 1-sentence rationale (e.g., "High growth potential + strong campaign engagement") |
| `risk_summary` | string | Account Health Agent output (risk factors) |
| `opportunity_summary` | string | Opportunity Agent output (growth signals) |
| `recommended_action` | string | Action Agent output (e.g., "Schedule 15-min optimization review call") |
| `action_type` | string | One of: `follow_up_call`, `reactivation_email`, `support_escalation`, `renewal_prep`, `expansion_offer` |
| `confidence_score` | float | Governance Agent output (0–1) |
| `evidence` | array | List of evidence objects (see below) |
| `governance_caveats` | array | List of caveat strings (e.g., "Limited recent engagement data") |
| `governance_status` | string | One of: `approved`, `pending_review`, `insufficient_evidence` |
| `draft_email` | string | Communication Agent output (seller-ready email) |
| `call_script` | string | Communication Agent output (talking points) |
| `voice_summary` | string | Communication Agent output (1-sentence seller brief) |
| `approval_status` | string | One of: `pending`, `approved`, `rejected` (default: `pending`) |
| `score_breakdown` | object | Sub-scores used in priority calculation |
| `agents_invoked` | array | List of agent names that processed this account |

### Evidence Object Schema

Each evidence item tracks the provenance of a signal:

| Field | Type | Description |
|-------|------|-------------|
| `source_agent` | string | Which agent surfaced this (e.g., `AccountHealthAgent`, `OpportunityAgent`) |
| `label` | string | Short tag (e.g., `spend_decline`, `campaign_click`, `support_ticket`) |
| `detail` | string | Human-readable description |
| `source_system` | string | Origin (CRM, Billing, Support, Telemetry, Marketing) |
| `polarity` | string | `positive`, `negative`, or `neutral` |
| `strength` | float | Signal weight (0–1) |

**Example:**
```json
{
  "source_agent": "AccountHealthAgent",
  "label": "spend_decline",
  "detail": "MRR decreased 15% month-over-month",
  "source_system": "Billing",
  "polarity": "negative",
  "strength": 0.85
}
```

## Schema: Decision Ledger

Written to SQLite `decisions.db` for every workflow invocation. Provides a tamper-evident audit trail.

| Field | Type | Description |
|-------|------|-------------|
| `ledger_id` | string | Unique ledger entry ID (UUID) |
| `timestamp` | string | ISO 8601 timestamp of workflow start |
| `user_query` | string | Original request (e.g., "Which SMB accounts need attention?") |
| `agents_invoked` | array | List of agent names executed (all six in sequence) |
| `evidence_used` | array | Aggregated evidence items across all recommendations |
| `reasoning_summary` | string | High-level workflow summary (e.g., "Analyzed 50 accounts, scored deterministically, identified 5 high-priority...") |
| `confidence_score` | float | Average confidence across recommendations |
| `caveats` | array | Aggregated governance caveats |
| `final_recommendation` | string | Executive summary for seller (top 3 actions) |
| `approval_status` | string | Overall status: `pending`, `partially_approved`, `fully_approved` |
| `steps` | array | Execution trace (agent name, latency, status) per step |
| `model_provider` | string | Which adapter was used (`mock`, `nvidia-nim`) |
| `latency_ms` | int | Total workflow latency in milliseconds |

## Data Generation

**Script:** `services/api/data/generate_synthetic_data.py`

**Process:**
1. Generate 50 accounts with randomized but realistic distributions:
   - Spend: $500–$15,000/month (SMB tier)
   - Renewal dates: -30 to +180 days (includes at-risk overdue renewals)
   - Scores: normal distribution around 50, clamped 0–100
2. Generate 200+ signals with temporal clustering (recent activity weighted higher)
3. Generate 75 notes with realistic seller observations
4. Write to `accounts.csv`, `signals.csv`, `notes.csv`

**Reproducibility:** Seeded random generation ensures consistent test data across runs.

## Privacy & Compliance

- **No real data:** All account names, industries, and signals are synthetic
- **No PII:** No email addresses, phone numbers, or personal identifiers
- **No confidential IP:** No references to real company strategies, pricing, or roadmaps
- **Hackathon-safe:** Designed for public demo and open-source release
